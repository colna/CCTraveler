# CCTraveler v0.3 当前实现与待实现

更新时间：2026-04-24

这份文档用于梳理目前仓库里已经完成的内容、尚未完成的内容，以及接下来建议的推进顺序。状态判断基于当前代码，而不是只看 `docs/ROADMAP_v0.3.md` 的原始规划。

v0.3 的核心功能已基本完成：

1. 完整城市地理数据接入已经完成。

3. 机票持久化与真实抓取闭环已完成。

## 二、已实现

### 1. 完整城市地理数据接入

这一阶段已经完成，不再是 roadmap 里的待办。

已落地内容：

- 新增正式 migration：`crates/storage/migrations/002_add_geo_lookup_tables.sql`
- 数据库初始化会自动创建 geo lookup 相关表：
  - `city_mappings`
  - `station_codes`
  - `airport_codes`
- 导入脚本已支持完整初始化与导入：`scripts/import_geo_data.py`
- Rust storage 层已经支持城市解析与地理查询。
- Python scraper 侧已经统一改为从 SQLite 做城市 / 车站 / 机场 lookup。
- `/cities` 接口已经切换为数据库驱动，而不是静态文件读取。

这意味着“完整城市地理数据接入”这一阶段已经从规划变成了现状。

### 2. 完整城市数据现在放在哪里

当前完整城市相关数据主要分成三层：

基础源数据：

- `data/ctrip_cities.json`：完整城市目录，当前用于导入 2173 个城市基础映射。
- `data/geo/cities.json`
- `data/geo/districts.json`
- `data/geo/attractions.json`
- `data/geo/stations.json`
- `data/geo/airports.json`

导入后的本地数据库：

- `data/cctraveler.db`

运行时读取入口：

- Rust 侧通过 storage 查询数据库
- Python 侧通过 `services/scraper/src/utils/geo_lookup.py`

也就是说，完整城市数据最终是以 SQLite 的形式被程序消费的，JSON 主要是导入源。

### 3. 新用户是否需要重新拉城市数据

代码仓库里的 JSON 数据文件是跟仓库一起走的，但 `data/cctraveler.db` 是本地运行态数据库。

所以对新用户来说：

- 如果仓库里没有现成可用的 `data/cctraveler.db`，需要执行一次导入脚本，把 JSON 导入到本地 SQLite。
- 如果已经提供了可直接使用的 `data/cctraveler.db`，则不需要重新拉原始城市数据，只需要直接使用这个数据库。

更准确地说，不是每次启动都要重新拉，而是“初始化本地环境时至少要准备一次数据库”。

### 4. 火车票最小闭环已经打通

这一部分已经明显超过 roadmap 里原来的“只有 mock”的状态。

当前已经实现：

- Rust storage 层新增 train 写入与查询能力：
  - `upsert_train(...)`
  - `insert_train_price(...)`
  - `search_trains(...)`
- `search_trains` 工具已经改成“先查库，再抓取，再落库”。
- 已支持按路线 + 日期查询最近快照缓存。
- Python scraper 的 `/scrape/trains` 已不再写死 mock，而是走可切换模式：
  - `mock`
  - `auto`
  - `real`
- `services/scraper/src/train/fetcher.py` 已经有最小真实抓取实现。
- 在开发验证里，已经验证过：
  - 第一次查询会抓取并落库
  - 第二次同条件查询会命中本地缓存

这说明“数据库查询逻辑”在 train 这条链路上已经不是空白，而是已经有一套最小可工作的实现。

### 5. 现有 train 能力的边界

虽然 train 已经有最小闭环，但当前稳定可用的状态是：

- 本地缓存闭环可用
- mock 模式可用
- real 模式已接通入口，但真实 12306 结果不稳定

所以它不是“完全完成”，而是“基础链路已完成，真实数据稳定性未完成”。

## 三、部分实现 / 尚未完全完成

### 1. 真实 12306 接入只完成了第一步

当前已经有：

- Selenium 入口
- 站码 lookup
- 页面加载与基本解析逻辑
- `auto/real/mock` 模式切换

但还没真正完成 roadmap 里定义的目标，主要缺口有：

- 还没上 `undetected-chromedriver`
- 没有验证码处理
- 页面风控下缺少稳定降级策略
- DOM 解析仍然偏脆弱
- 当前环境下会被 12306 风控页拦截，不能认为已经达到“真实接入完成”
- 成功率远达不到 roadmap 里的稳定性目标

所以 Phase 2 现在的真实状态应该是：

- 已完成最小闭环
- 未完成生产级真实 12306 接入

### 2. 车站默认选择策略还不够好

当前 `get_station_code(city)` 的策略是“匹配到城市后取一个站”。这会带来一个实际问题：

- 同一个城市有多个车站时，默认命中的站不一定是用户真正想查的主站。

已经观察到的典型情况是：

- “北京”会被解析到 `VAP`
- “上海”会被解析到 `SNH`

这对真实 12306 查询是有影响的，所以这块还需要单独修。

### 3. 机票仍然主要是 mock

机票这条链路目前还停留在 roadmap 早期阶段。

当前状态：

- `search_flights` 工具存在
- Python 侧 mock 数据存在
- `fetch_flights_ctrip(...)` 还是空框架
- 还没有多源聚合
- 还没有机票持久化 / 本地缓存闭环

也就是说，flight 目前明显落后于 train。

### 4. compare_routes 仍然是 mock

`compare_routes` 目前还是示例性返回，没有真正调用已实现的 train / flight 数据链路。

未完成部分包括：

- 真实调用 train 查询
- 真实调用 flight 查询
- 并发查询优化
- 基于真实数据做时间 / 价格 / 舒适度评分
- 输出真正可用的路线推荐结果

## 五、当前最大的阻塞

### 1. 12306 风控

这是目前最明显的真实数据阻塞点。

现状不是代码完全没写，而是：

- 已经有真实抓取尝试
- 但 12306 在当前环境下会返回风控 / 错误页
- 所以现在还不能把这条链路当成稳定生产能力

### 2. 城市到默认车站的映射策略

即使绕过了风控，这块也会继续影响查询质量。

当前更合理的方向应该是：

- 城市默认站策略更精细
- 或者直接支持“城市 -> 候选站列表 -> 选择主站 / 优先站”
- 避免简单按字典序或首条命中返回
